These recipes chain Scram's auth actions into common flows. They come in **issue / consume** pairs: one recipe issues a token and sends the email, a matching recipe handles the click.

<Note>
  The raw token only ever travels by email. Everywhere else it is hashed. Email possession proves control of the inbox, and hashing means a leaked database does not leak usable tokens.
</Note>

<Warning>
  **Where the email actually goes.** These recipes send through the **Send Email** reusable workflow. While that workflow uses the built-in **Send Email to Project Owner** action, every message is delivered to _your_ inbox â€” the project owner's verified address â€” whatever the **to** input says. That is by design for testing: you receive the token email even for a user you just invited, and can click through the whole flow yourself. Before real users need to receive these emails, swap the send step inside the workflow to your email provider's HTTP data source. See [Sending email: the test sender and going live](/recipes/sending-email).
</Warning>

## How the recipes fit together

Recipes 1 and 6 **issue** tokens. Recipes 2, 3, and 4 **consume** them. Recipe 5 stands alone â€” no email involved.

| Recipe | Issues | Consumes | Signs user in |
| --- | --- | --- | --- |
| 1. Send magic link / reset email | Yes | â€” | â€” |
| 2. Magic-link login | â€” | Yes | Yes |
| 3. Password reset | â€” | Yes | No |
| 4. Email verification | â€” | Yes | No |
| 5. Change password (signed in) | â€” | â€” | Already in |
| 6. Invite / passwordless signup | Yes | via recipe 2 | via recipe 2 |

All token-carrying recipes share one storage slot on the `Users` table: `passwordResetToken` and `passwordResetTokenExpiresAt`. There is no separate column for magic-link, verification, or invite tokens â€” see [the shared token slot](#the-shared-token-slot) for what that means in practice.

## 1. Send a magic link or reset email

Issues a token to a user who **already exists** (e.g. bulk-loaded, or created by an admin process) and emails it. Always reports success to the frontend so nobody can probe which emails exist.

<Note>
  This recipe assumes the user already exists. To create the user at signup time instead, see [recipe 6](#6-invite--passwordless-signup), which uses an `INSERT`.
</Note>

<Steps>
  <Step title="Generate the token">
    Backend server request block. `Generate Secret Token` to mint the raw token.
  </Step>
  <Step title="Store the hash and detect whether a user matched">
    `UPDATE Users` setting `passwordResetToken` to the hashed token and `passwordResetTokenExpiresAt` to `NOW() + INTERVAL '1 hour'`, `WHERE email = :email`, with `RETURNING id`. A matched row returns the id; no match returns an empty result. **This is the check** â€” an `UPDATE` alone reports success even when it matches zero rows, so use `RETURNING` to know whether a user was actually found.

    Always write the expiry in the same statement as the token. A token stored with a null expiry is the failure case the consume recipes are written to reject â€” see [expiry and the null case](#expiry-and-the-null-case). An hour suits password resets and magic links; invites want longer (recipe 6).
  </Step>
  <Step title="Branch on the result">
    `Switch Case (Backend)` on whether the returned result is empty.
  </Step>
  <Step title="Matched: send the email">
    Call the **Send Email** reusable workflow with the user's address, a subject, and a body containing the link with the raw token.
  </Step>
  <Step title="No match: skip the email">
    Do not send an email â€” there is no user to send to. Log or flag the miss if you want visibility on issues sent to unknown addresses. Do **not** treat this as success internally just because the `UPDATE` returned green.
  </Step>
  <Step title="Return uniform success to the frontend">
    Regardless of branch, the frontend always sees success â€” so nobody can probe which emails exist. The distinction between matched / no-match stays server-side.
  </Step>
</Steps>

<Warning>
  An `UPDATE ... WHERE email = :email` that matches no rows reports **Success** and returns an empty result â€” it does not error. Without `RETURNING id` and a branch on the empty result, a link issued to a non-existent user silently produces a live token with no stored hash, which then fails at consume with no signal upstream. Always branch on the `RETURNING` result.
</Warning>

## 2. Log in via magic link

Consumes the token from recipe 1 and signs the user in.

<Note>
  `Login With Token` compares whatever you pass it against exactly what is stored. Because recipe 1 stores the token hashed, you must hash the URL token before passing it. (If you stored the token raw, you would pass it raw â€” hash on consume only if you hashed on store.)
</Note>

<Steps>
  <Step title="Hash the URL token">
    Backend server request block hashes the token from the URL.
  </Step>
  <Step title="Consume via Login With Token">
    Frontend calls `Login With Token` with the email and hashed token. `Login With Token` requires both the email and the token, so the link must carry both.
  </Step>
  <Step title="Branch on result">
    Success navigates to the target path. Failure shows "link invalid or expired".
  </Step>
</Steps>

<Warning>
  Run `Login With Token` in a context that can act on its result â€” not in a page-data workflow. Page-data can redirect on success but cannot cleanly surface the failure, so a failed login falls through to the success path. `Login With Token` returns `success`, `errorCode`, and `message`. Branch on `success`, and display `message` â€” but note it may come back as an **empty string** (not null) on failure, so use `||` rather than `??` for your fallback text, or a blank message renders as nothing.
</Warning>

## 3. Password reset (no auto sign-in)

Consumes a reset token and sets a new password. Does not sign the user in.

<Steps>
  <Step title="Validate the token">
    Backend server request block validates the email, the hashed token, and the expiry in a single `WHERE` clause:

    ```sql
    WHERE "email" = :email
      AND "passwordResetToken" = :hashedToken
      AND "passwordResetTokenExpiresAt" > NOW()
    ```

    Write the expiry check in that direction. A null expiry then fails the comparison and the token is rejected â€” see [expiry and the null case](#expiry-and-the-null-case).
  </Step>
  <Step title="Branch">
    `Switch Case (Backend)`.
  </Step>
  <Step title="Set the password">
    `Set Password`. Its defaults clear the token and revoke existing sessions.
  </Step>
  <Step title="Redirect to login">
    Frontend navigates to `/login` with a success indicator (e.g. a query param that surfaces a message). On error, show the error.
  </Step>
</Steps>

## 4. Email verification (no auto sign-in)

Consumes a verification token and marks the email verified.

<Steps>
  <Step title="Atomic verify">
    Backend server request block runs one atomic `UPDATE` setting `emailVerifiedAt` and clearing both token columns, matching on email, hashed token, and expiry, with `RETURNING id`:

    ```sql
    UPDATE "Users"
       SET "emailVerifiedAt" = NOW(),
           "passwordResetToken" = NULL,
           "passwordResetTokenExpiresAt" = NULL
     WHERE "email" = :email
       AND "passwordResetToken" = :hashedToken
       AND "passwordResetTokenExpiresAt" > NOW()
    RETURNING id
    ```
  </Step>
  <Step title="Branch on result">
    `Switch Case (Backend)` on success / error.
  </Step>
</Steps>

## 5. Change your own password (signed in)

No email involved. The user is already signed in.

<Steps>
  <Step title="Validate current password">
    `Validate Password` with the user id and the current password.
  </Step>
  <Step title="Branch">
    `Switch Case (Backend)`. On failure, show "current password incorrect".
  </Step>
  <Step title="Set the new password">
    `Set Password` with the new password.
  </Step>
</Steps>

## 6. Invite / passwordless signup

The user does not exist yet. The account is created at invite time.

<Steps>
  <Step title="Look up the email">
    Backend server request block looks up the email.
  </Step>
  <Step title="Branch">
    `Switch Case (Backend)`.
  </Step>
  <Step title="Not found: create the account">
    `Generate Secret Token`, then `INSERT` into Users: email, `authType` `userpass`, `roles` `ARRAY[]::text[]`, `passwordResetToken` = hashed token, `passwordResetTokenExpiresAt` = now \+ expiry. Use a longer window than a password reset â€” invites often last days. Then call the **Send Email** reusable workflow with the raw token.
  </Step>
  <Step title="Found: no-op or re-issue">
    Do nothing, or re-issue.
  </Step>
  <Step title="Return uniform success">
    The frontend always sees success.
  </Step>
</Steps>

The invited user consumes the link with **recipe 2** (`Login With Token` signs them in). The new account has no password.

<Warning>
  The invited user is created with no roles (`ARRAY[]::text[]`). They can sign in via the magic link, but will be blocked from any auth-required page or workflow that has roles selected until a role is assigned. Make sure your invite flow assigns a role, or the user signs in and sees nothing.
</Warning>

To let the user log in with a password later, send them to a **Set Password** page after first sign-in and call `Set Password` directly.

<Note>
  `Set Password` takes `userId` and the new password (plus `Clear reset token` and `revoke sessions` flags). It sets the password unconditionally â€” it does not check any current password â€” so it works whether the row was created with no password or with an unknown one. The current-password check in recipe 5 comes from the separate `Validate Password` action, not from `Set Password`. This is why the invite flow needs no password workaround: `Set Password` simply sets one.
</Note>

## The shared token slot

Every token-carrying recipe on this page writes to the same two columns: `passwordResetToken` and `passwordResetTokenExpiresAt`. There is no dedicated magic-link, verification, or invite token field.

<Warning>
  Because the slot is shared, **two live token flows for the same user overwrite each other**. Issuing a password reset invalidates a pending verification link, and vice versa.

  The expiry is overwritten too. A reset issued with a one-hour expiry replaces the deadline on an invite that was issued with a seven-day one â€” so the earlier link does not merely stop working, it stops working on a schedule its own flow never set.
</Warning>

In practice this is fine for signup-time verification and for invites, where the flows do not realistically overlap. Take care if you add post-signup re-verification, where a user could plausibly request a password reset while a verification link is outstanding. If you need both live at once, add your own columns rather than reusing this slot.

## Expiry and the null case

The consume recipes match on `"passwordResetTokenExpiresAt" > NOW()`. That direction matters.

Postgres comparisons against `NULL` evaluate to `NULL`, not `TRUE`, so a row whose expiry was never written fails the `WHERE` clause and the token is rejected. The check fails closed.

<Warning>
  Do not rewrite the expiry check as a negation such as `NOT ("passwordResetTokenExpiresAt" < NOW())`, and do not drop the clause when the column might be null. Both read as "not expired" in English, but a null expiry then passes, producing a token that never expires. Always compare the expiry directly and in the positive direction.
</Warning>

The other half of this is on the issue side: always write the expiry in the same statement that writes the token (recipes 1 and 6). A token stored without one is unusable under the recipes above â€” which is the safe outcome, but a confusing one to debug if you did not intend it.
