Events are how components signal that something has happened. You attach a workflow to an event in the **Logic tab** of the Component Editor — when that event fires, the workflow runs.

The events available depend on the component. Not every component supports every event.

---

## Available Events

| Event | Description |
| --- | --- |
| `click` | The user clicked the component |
| `doubleClick` | The user double-clicked the component |
| `contextMenu` | The user right-clicked the component |
| `mouseEnter` | The pointer moved onto the component |
| `mouseLeave` | The pointer moved off the component |
| `blur` | The component lost focus |
| `everyChange` | Fires on every keystroke or value change — useful for live search |
| `committedChange` | Fires when the user commits a value — pressing Enter on an input, selecting a chip |

---

## Event Payloads

Every event passes a payload to the triggered workflow, accessible via `trigger.payload`.

### MinimalEvent

Used by `mouseEnter`, `mouseLeave`, and `blur`.

```js
{
  timestamp: new Date(),
  componentId: component.id,
  componentName: component.name
}
```

### PointerEvent

Used by `click`, `doubleClick`, and `contextMenu`.

```js
{
  x: trigger.payload.clientX,
  y: trigger.payload.clientY,
  keyAlt: trigger.payload.keyAlt,
  keyCtrl: trigger.payload.keyCtrl,
  keyMeta: trigger.payload.keyMeta,
  keyShift: trigger.payload.keyShift,
  timestamp: new Date()
}
```

### Change events

For `everyChange` and `committedChange`, the payload carries the new and previous values:

```js
trigger.payload.newValue   // the current value
trigger.payload.oldValue   // the previous value
```

<Note>
  These payloads are derived from standard HTML events but simplified in some cases. The underlying browser event data is the source.
</Note>

## Debouncing event workflows

Some events fire very frequently — `everyChange` on a text input fires on every keystroke, `mouseMove` fires constantly as the pointer moves. Running a full workflow on every one of those firings is wasteful and can make your app feel sluggish, especially if the workflow hits the database.

Every workflow has a **debounce** setting in its **Workflow Config**. It delays the workflow from running until the triggering event has stopped firing for a specified number of milliseconds.

With a debounce of `300ms`, a workflow attached to `everyChange` on a search input will wait until the user pauses typing for 300ms before running — once, with the final value — rather than running on every keystroke.

### When to use it

| Scenario | Typical debounce |
| --- | --- |
| Search-as-you-type against the database | 250–400ms |
| Auto-save on a form | 500–1000ms |
| Live filtering of an in-memory list | 100–200ms |
| Resizing a chart on window resize | 100–200ms |

<Tip>
  For most typeahead and search inputs, 300ms is a good default — long enough to avoid firing on every key, short enough that the user doesn't notice the wait.
</Tip>

<Note>
  Debounce is configured per workflow, not per event. The same event can have one workflow that fires immediately and another that fires only after the user pauses, if that's useful.
</Note>

## Custom Component Events

Custom components can emit their own events using the **Emit Event** action. The parent component or page listens for these in its Logic tab, just like platform component events.

See [Building Your Own Components](/understanding/subpages/building-your-own-components) for how to define and emit custom events, and [Understanding Events and Workflows](/understanding/understanding-events-and-workflows) for how event propagation works in Scram.
