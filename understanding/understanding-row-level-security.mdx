Table Security (often known as Row Level Security - RLS) is how Scram controls which users can read, write, update, or delete which rows in your database â€” enforced at the database level, not just in your app logic.

---

## The core idea

Without table security, any query can read or modify any row. With it, every query is automatically filtered based on who is making the request. A user asking for their orders will only ever get their orders â€” the database itself enforces this, regardless of how the query is written.

<Note>
  In Scram, every table you create must have at least one security policy before it can be used.
</Note>

---

## Default behaviour

When you ask Scram AI to create a new table, it will automatically apply a sensible default policy based on what the table is likely to be used for. A table for blog posts or products will typically be open for anyone to read, while a table for accounts, orders, or user-specific records will default to creator-only access. In most cases, Scram AI makes the right call without any extra instruction â€” but you can always describe the behaviour you want and it will adjust accordingly.

---

## Policy types

**Permissive (Allow)** â€” grants access. At least one must exist on every table. If multiple permissive policies exist, a row is accessible if any one of them passes.

**Restrictive (Deny)** â€” narrows access on top of permissive policies. A row must pass all restrictive policies as well as at least one permissive policy to be accessible. Use these to layer additional constraints â€” for example, blocking access to suspended accounts even if the base policy would allow it.

---

## What policies apply to

Each policy targets a specific SQL operation:

| Command | What it covers |
|---|---|
| `SELECT` | Reading rows |
| `INSERT` | Creating new rows |
| `UPDATE` | Modifying existing rows |
| `DELETE` | Removing rows |
| `ALL` | All of the above |

---

## USING vs WITH CHECK

Policies have two clause types:

**USING** â€” filters which existing rows are visible or affected. Applies to `SELECT`, `UPDATE`, `DELETE`, and `ALL`. Think of it as: "which rows can this user see or touch?"

**WITH CHECK** â€” validates rows being written. Applies to `INSERT`, `UPDATE`, and `ALL`. Think of it as: "is the row this user is trying to create or change allowed?"

The rule for which to use:

| Operation | Clause |
|---|---|
| `SELECT`, `DELETE` | `USING` only |
| `INSERT` | `WITH CHECK` only |
| `UPDATE`, `ALL` | Both |

---

## Auth functions

Three utility functions are available inside policy expressions:

```sql
auth.user_id()    -- The logged-in user's ID (matches Users.id)
auth.email()      -- The logged-in user's email address
auth.roles()      -- Array of the user's role IDs (text[])
```

These return `null` for unauthenticated requests.
