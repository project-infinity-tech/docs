---
title: "Understanding The Scram File Store"
---

Every Scram project has a File Store that is shared across all apps in your project. It's a built-in cloud storage system for files your app needs to handle at runtime — user uploads, images, documents, PDFs, and any other files users create or submit while using your app.

<Note>
  The File Store handles **runtime file uploads** — files that users upload while using your app. If you're looking to manage static design assets like logos, background images, or icons that you add while building your app, that's [Project Files](/understanding/understanding-project-files).
</Note>

---

## The Uploader Component

The uploader component exposes three properties. Two of them — **Path** and **Check for Duplicates** — determine where files go and whether uploads are allowed to collide, so they're worth understanding before you build.

| Property | Description |
| --- | --- |
| **Icon** | The icon shown on the uploader control. |
| **Path** | The base path files are uploaded to. Defaults to `uploads`. You can change it, and you can build it dynamically to control exactly where each file lands. |
| **Check for Duplicates** | Whether to block an upload when a file already exists at the target `path`. Defaults to **ON**. |

---

## Path Is The File's Identity

In the File Store, **the `path` is the file's identity** — it's the S3 object key. Whatever you set as the path _is_ where the file lives, and _is_ what determines whether two uploads collide. Almost every upload behaviour you'll want is achieved by controlling the path.

- **Fixed path (default).** With Path left as `uploads`, files land in one place keyed by filename. Two files named `cv.pdf` target the same key and collide — blocked or overwritten depending on Check for Duplicates.
- **Per-file unique path.** Build the path to include a unique token, e.g. `uploads/a1b2c3d4 — cv.pdf`. Every upload is distinct, nothing collides, and you keep all files.
- **Scoped path.** Include a scope in the path, e.g. `uploads/{jobId}/cv.pdf`, to make files unique _within_ that scope — useful for "one CV per applicant per job" structures.

<Note>
  S3 has no real folders. A path like `uploads/{jobId}/cv.pdf` is a single flat key that _contains_ slashes — it behaves like a folder for listing (you can list everything under `uploads/{jobId}/`) but there is no directory object. If you need a true directory structure — renaming folders, moving files, listing reliably by folder — store file records (id, path, original name, owner, parent) in a database table and treat the File Store as flat blob storage keyed by path.
</Note>

See the [File Upload Recipes](/recipes/file-uploads) for worked patterns covering keep-all, overwrite-in-place, and scoped uniqueness.

## Duplicate Handling

**Check for Duplicates** defaults to ON and controls whether the store is checked for an existing file before uploading.

- **ON (default).** Before upload, the File Store is checked for an existing file at the same `path`. If one exists, the upload is blocked and the user sees `A file with this name already exists`.
- **OFF.** Standard S3 behaviour applies — uploading to an existing key **overwrites** whatever is already at that `path`. No warning is shown.

<Note>
  The default is ON deliberately. With it off, an upload silently overwrites any existing file at the same `path` anywhere in the project's File Store — the more destructive default. Leave it on unless you specifically want overwrite behaviour, such as a user replacing their own profile picture at a stable path.
</Note>

<Warning>
  The check keys on `path`, not on file content. Two genuinely different files uploaded to the same `path` still collide. To keep multiple files that share a filename, make the `path` unique per file — see the recipes linked above.
</Warning>

## How Uploads Work

The File Store uses a three-step upload process.

### 1. Prepare File Upload

Requests a pre-signed upload URL for a file. This step runs **on the server**.

| Input | Description |
| --- | --- |
| `path` | Where to store the file (e.g. `uploads/photo.jpg`) |
| `contentType` | The file's MIME type (e.g. `image/jpeg`) |

| Output | Description |
| --- | --- |
| `uploadUrl` | A temporary pre-signed URL to upload to |
| `fileUri` | The permanent S3 identifier for the file |

### 2. Upload File to URL

Sends the actual file to the pre-signed URL. This step runs **in the browser**.

| Input | Description |
| --- | --- |
| `file` | The file object from the user's file picker |
| `uploadToUrl` | The `uploadUrl` returned from step 1 |

<Warning>
  Steps 1 and 2 run in different environments — server and browser. You must store `uploadUrl` in a page variable between these steps to bridge back to the browser context. Without this, you will see the error: `Step not found in given list of steps — likely a frontend step`.
</Warning>

### 3. List Files

Lists files stored at a given path.

| Input | Description |
| --- | --- |
| `path` | The folder path to list |

Returns an array of file objects with this shape:

```json
{
  "name": "filename.ext",
  "fileUri": "s3://bucket/path/to/file",
  "size": 1333,
  "lastModified": {}
}
```

<Note>
  `lastModified` always returns as an empty object — the storage system does not populate this field. Display `size` instead, formatted in KB by dividing bytes by 1000.
</Note>

## Implementation Notes

### Capturing the file

The file input's `onChange` event returns the full browser event object. The actual file lives at:

```js
trigger.payload.target.files[0]
```

Do not use the top-level trigger payload — it will send the raw event object to the upload step, which will fail with `file is required`.

### Filename and content type

By default, Scram saves every uploaded file with the hardcoded name `file` unless you explicitly pass the real filename in the upload path.

- Read the filename from the file object: `file.name`
- Read the content type from the file object: `file.type`

If `file.type` is unavailable, fall back to `application/octet-stream`. Do not hardcode a content type — it will cause MIME type mismatches for images, PDFs, and other formats.

### Building the upload path

Because the `path` is the file's identity, this is where you decide collision behaviour. A typical dynamic path combines the component's base Path with a unique or scoped segment and the real filename — for example `uploads/{jobId}/{token} — {file.name}`. Keep the construction consistent so you can strip any prefix back off for display. See the [File Upload Recipes](/recipes/file-uploads) for the exact patterns.

### Resetting the file input after upload

Browser file inputs cannot be reset with a simple value clear. The most reliable approach in Scram is to bind the input's `value` to a filename variable and clear that variable on successful upload. When the variable empties, the input resets visually.

### Refreshing the file list

The List Files action must be triggered on page load and re-triggered after each successful upload to stay current without a manual refresh.

## Common Pitfalls

| Issue | Cause | Fix |
| --- | --- | --- |
| `A file with this name already exists` | Check for Duplicates is on, and a file already exists at that `path` | Make the `path` unique per file, or turn Check for Duplicates off to overwrite |
| `Step not found` error | Server and browser steps running out of order | Store `uploadUrl` in a page variable to bridge back to browser context |
| Filename saves as `file` | Real filename not passed to the upload path | Read `file.name` directly from the file object |
| `[object Object]` in file list | Raw object rendered as text | Use the correct field accessor (`.name`, `.size`) |
| Content type mismatch | Hardcoded MIME type | Use `file.type` from the file object |
| File input doesn't reset after upload | Browser input limitation | Bind input value to a variable and clear it on successful upload |
| Date shows as `{}` | Storage system doesn't return `lastModified` | Don't display it — show `size` instead |
