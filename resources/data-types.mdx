The Data Types section of the Resource View gives you visibility into all the types available in your project — from types you define yourself to built-in platform and editor types. Understanding the type system helps you build more robust components, write cleaner expressions, and catch errors earlier.

---

## Project Types

Project Types are custom types you define yourself. These are the types you'll create most often when modelling your app's data.

Each type is made up of one or more fields, and each field has a type of its own — which can be a primitive (text, boolean, integer) or any other type in the system, including other Project Types.

Fields can also have **validators** attached — rules that constrain what values are acceptable. Validators vary by field type. For example, a text field might use `isEmail` or `isURL`, while a number field might use `isInteger`. A full validator reference is available in the Reference section.

New projects start with no Project Types. Click **Add Type** to create one, then add fields and validators as needed.

---

## Database Types

Every table in your Scram Database automatically gets a corresponding Database Type — a read-only type that reflects the table's schema. These are generated by Scram and cannot be edited here. To change a Database Type, update the underlying table schema in the [Database](/resources/database) section.

---

## Platform Namespaces

Platform Namespaces are built-in system types provided by Scram. They represent common data concepts that would otherwise require you to create a Project Type with a text field and a validator manually.

Examples include:

- `Email` — a valid email address
- `URL` — a valid URL
- `UUID` — a universally unique identifier
- `Base64` — Base64-encoded data

Use Platform Namespace types wherever you'd otherwise create a plain text type just to apply a standard validator.

---

## Editor Types

Editor Types are types used within the Scram editor and component system — some are editor-only, others are available at runtime too. They define things like component sizes, icon names, and audio files.

Examples include `Size` (an enum used to control component sizing), `IconName`, `AudioFile`, and `AuthProviderType`. These are provided by Scram and cannot be edited.

---

## Style Types

Style Types are CSS-related types and enums used to control the visual properties of components. They map directly to CSS concepts.

Examples include:

- `Colour` — a text value representing a colour
- `CssAlignItems` — an enum: `start`, `centre`, `end`
- `CssJustifyContent` — an enum: `start`, `center`, `end`, `space-between`, `space-around`, `space-evenly`

---

## Event Payloads

Event Payloads are the standard types that Scram components emit when they fire events. They carry contextual data about what triggered the event.

For example, a **PointerEvent** — fired on click — includes:

- `keyCtrl` — whether the Ctrl key was held
- `keyShift` — whether the Shift key was held
- `x` / `y` — the pointer coordinates
- `timestamp` — when the event occurred

A full Event Payload reference is available in the Reference section.
