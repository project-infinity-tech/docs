# Generate Secret Token

# Generate Secret Token — Workflow Action

**Generate Secret Token** mints a cryptographically secure 256-bit random token for use in a magic-link, password-reset, or email-verification flow. It is the starting point of any flow that issues a one-time link by email.

---

## What is Generate Secret Token?

It is a **backend-only workflow step** that produces a single random token. The token is the credential a user later presents by clicking a link in an email. The action itself does nothing with the token beyond returning it — persisting it and emailing it are separate steps you add after it.

The critical property is that the output is **restricted**: only later backend steps **in the same Server Request Block** can read the token. It never reaches the browser.

---

## Output

A single value: the raw token.

Because the output is restricted, it is readable only by subsequent backend steps in the same Server Request Block. This is the constraint that shapes how you build an issue flow — the steps that generate, store, and email the token must all live in the **same** Server Request Block, or a later step will not be able to read the token it needs.

---

## How it Works at Runtime

1. The step runs and generates a 256-bit random token.
2. The raw token becomes available to later backend steps in the same Server Request Block.
3. Those later steps persist a **hash** of the token and email the **raw** token to the user.

The raw token exists server-side and in the email only. It is never stored and never sent to the frontend.

---

## Common Patterns

### Issuing a link

```text
Step 1: Generate Secret Token          → raw token
Step 2: UPDATE Users SET token = hash  → persist encode(sha256(:token::bytea), 'hex')
Step 3: Send Email                     → email the raw token in the link
```

All three steps live in one Server Request Block so step 2 and step 3 can read the token from step 1.

---

## Important Constraints

- **Backend only.** It cannot be run from the frontend.
- **Output is restricted to the same Server Request Block.** Generate, store, and send must be in one block.
- **Never store the raw token.** Persist a hash (`encode(sha256(:token::bytea), 'hex')`); the raw token travels only in the email. A leaked database of hashes cannot be used to log in.

---

## Relationship to Other Actions

- **Login With Token** — consumes the token this action issues.
- **Send Email** — delivers the raw token to the user.

See [Auth flow recipes](/recipes/auth-flow-recipes) for complete issue and consume flows.

---

In summary: Generate Secret Token is the **mint** step of any email-link auth flow. It produces the credential; storing its hash and emailing the raw value are the steps you build around it.
