# Set Password

# Set Password — Workflow Action

**Set Password** sets a user's password. It is used at the end of a password-reset flow, when an invited user chooses their first password, and whenever an admin needs to set a password directly.

---

## What is Set Password?

It writes a new password for a user, identified by id. It is **unconditional**: it does not check any current password. It works the same whether the user currently has no password, a password they know, or a password they don't know.

This is the key thing to understand — the "do you know your current password?" check is **not** part of Set Password. If you need that check, you run Validate Password first (see below). If the flow has already been authorised another way — by a reset token, or by an invite link — you call Set Password directly.

---

## Inputs

| **Input** | **Type** | **Notes** |
| --- | --- | --- |
| **userId** | text | The id of the user whose password is being set. |
| **newPassword** | text | The new password. |
| **Clear reset token** | boolean | When set, clears the stored token so it cannot be reused. Use after a reset or invite consume. |
| **revoke sessions** | boolean | When set, revokes the user's existing sessions. |

---

## How it Works at Runtime

1. It sets the password for the user with the given `userId`.
2. If **Clear reset token** is set, the stored token is cleared in the same operation.
3. If **revoke sessions** is set, the user's other sessions are invalidated.

There is no current-password comparison at any point. Authorisation to call it comes from the surrounding flow, not from the action.

---

## Common Patterns

### Completing a password reset

```text
Step 1: validate email + hashed token + unexpired
Step 2: Set Password(userId, newPassword)   → Clear reset token: true, revoke sessions: true
```

The reset token authorised the change, so Set Password is called directly.

### Letting an invited user set their first password

```text
Step 1: user is signed in via Login With Token (invite link)
Step 2: Set Password(userId, newPassword)
```

Works even though the invited user never had a password.

### Changing your own password (with a current-password check)

```text
Step 1: Validate Password(userId, currentPassword)
Step 2: branch → if valid, Set Password(userId, newPassword)
```

Here the check comes from Validate Password, not Set Password.

---

## Important Constraints

- **Unconditional.** No current-password check — gate it yourself with Validate Password when that check matters.
- **Identified by userId, not email.** Resolve the user to an id first.
- **Use Clear reset token after token flows** so a consumed link cannot be replayed.

---

## Relationship to Other Actions

- **Validate Password** — run first when you need to confirm the user's current password before setting a new one.
- **Login With Token** — an invited user is signed in by Login With Token, then sets a password with Set Password.

See [Auth flow recipes](/recipes/auth-flow-recipes) for complete flows.

---

In summary: Set Password is the **write** step for a user's password. It sets unconditionally, so it fits reset, invite, and admin flows directly — and pairs with Validate Password when you need a current-password check first.
